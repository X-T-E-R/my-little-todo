# 核心辩论与设计决策

> 这份文件记录设计过程中每一个有争议的点、双方论据、以及最终（或暂定）的决策。每个决策都标注了状态：✅ 已决定 / 🔶 暂定 / ❓ 待讨论。
>
> 新的辩论应追加在最后，保留时间线。

---

## 辩论 #1：todo 工具为什么会被用户抛弃？

**起因**：在讨论"怎么做一个好 todo"之前，先理解为什么现有的都失败了。

**论点 A（表面原因）**：功能不够好、不够方便、不够跨平台。
→ 反驳：Todoist、Things 3、滴答清单都做得很好了，用户照样弃用。

**论点 B（深层原因）**：todo 工具会变成一面镜子，照出你所有没兑现的承诺。每次打开都是一个微小的心理惩罚。焦虑累积到阈值，用户选择逃避。

**决策 ✅**：产品必须从根源上避免"未完成焦虑"。默认视图不呈现完整的未完成列表，而是聚焦于"现在可以做的一件事"。

## 辩论 #2：输入应该"零摩擦"吗？

**论点 A**：捕捉越快越好，零摩擦才能保证"每个念头都被记录"。所有文献和用户直觉都支持这一点。

**论点 B（反面）**：输入越容易，堆积越多。一个月后 inbox 里 300 件未处理的东西，inbox 本身变成焦虑源。所以某些环节需要刻意的摩擦。

**综合决策 🔶**：**非对称摩擦设计**——
- 捕捉（往流里倒想法）：零摩擦
- 承诺（给任务设 DDL、进入"今天做"）：有一点点仪式感和确认
- 延期/删除承诺：有轻微但有效的摩擦（写理由）

这样，进入系统很容易，但"对自己承诺"这个动作有重量。

## 辩论 #3："此刻"视图——谁来决定现在做什么？

**矛盾的本质**：
- 用户自己选 → 决策成本高 → 违反"低认知成本"宪法条款
- AI 替用户选 → 自主性丧失 → 违反 SDT（自我决定理论）

**关键洞察**：用户需要的不是"最优推荐"，而是"有一个外部力量替我做了选择"。准确性是次要的，"替我选了"才是核心价值。

**决策 ✅**：AI 给出强推荐（一个任务 + 理由），用户可以接受或拒绝。拒绝时提供简单的原因选项（系统学习用）。同时提供 🎲 随机模式——从今天可做的任务里纯随机选一个，满足"随便给我一个就行"的需求。

## 辩论 #4：游戏化奖励该不该做？

**支持游戏化的理由**：
- 增加趣味性和使用动力

**反对游戏化的理由（过度辩证效应）**：
- 对内在动机的事加外在奖励，会侵蚀内在动机（Deci & Ryan）
- 连续打卡一旦中断 → "缺少奖励"本身成为心理惩罚
- XP/等级/徽章会把"完成有意义的事"变成"刷分"

**决策 🔶**：不做传统游戏化（积分/等级/连续天数）。做三件事替代——
1. **即时完成感**：精心设计的完成动效/音效（感官层面）
2. **行为洞察作为奖励**：周期性的"关于你的发现"（认知层面）
3. **进度可视化**：项目维度的进展，而非任务计数（意义层面）

状态为"暂定"，因为需要原型测试后再验证。

## 辩论 #5：产品形态——对话优先 vs 列表优先 vs 混合

**三条路线**：

| | 对话优先 | 列表优先 | 混合 |
|---|---|---|---|
| 输入体验 | 极好（自然语言） | 一般（填表单） | 好 |
| 浏览/回顾体验 | 差（线性、难扫视） | 好（结构化） | 好 |
| 差异化 | 极强 | 弱 | 中 |
| 风险 | 对话 vs 结构的冲突 | 和现有产品同质化 | 两头不精 |

**决策 ✅**：**混合模式**——
- 输入通道：对话流（Stream），随手记录
- 消费/执行通道：结构化视图（此刻/承诺板）
- AI 做翻译：把流中的内容提取为结构化任务

关键设计挑战：流和结构之间的映射必须可靠且可解释。用户要能理解"我写的那条→变成了这个任务"的对应关系。

## 辩论 #6："此刻"视图应该多强势？

**讨论**：用户明确需要外部推力，且能以幽默的方式接受"被强制"。但强制过头会产生抵触。

**决策 ✅**：
- 开屏默认进入"此刻"视图
- 可以划走切换到其他视图（流/承诺板）
- 但划走 3 次后，需要强制面对"此刻"一次（温和但坚定）
- 强制时的语气不是"你必须做事"，而是"嘿，先看一眼？"
- 如果真的不想做任何事，可以选"现在不是工作时间"→ 系统放行，不再打扰直到下一个时段

## 辩论 #7："不执行建议"到底意味着什么？

**传统思路**：用户没按计划做 = 计划失败 = 需要更好的计划/提醒/惩罚。

**颠覆性观点**：用户的行为（做/不做/拖延/逃避）不是对系统的"违规"，而是一种沟通。每一次不服从都在说："你的建议不够了解我。"

**这改变了整个产品的认识论。**

**决策 ✅**：构建"学习引擎"——
- 拒绝推荐时收集轻量原因（4 选 1，不超过 1 次点击）
- 追踪行为模式（启动时间、回避类型、持续时长、DDL 偏差）
- 用模式自动调整推荐策略（而非要求用户修改设置）
- 周期性地把洞察反馈给用户（"你是慢热型选手"）

## 辩论 #8：角色/项目切换——过滤器 vs 心理空间

**背景调研**：
- 上下文切换研究：人每天平均切换上下文 1200 次，每次切换后需要约 23 分钟恢复专注（注意力残留效应）
- PARA 方法（Tiago Forte）：每个角色对应一组"责任领域（Area of Responsibility）"，项目会结束，但角色是持续的
- 核心洞察：角色切换不仅仅是任务过滤，它是**心智模式的切换**

**讨论的五个方向**：

| 方向 | 核心思路 | 优点 | 缺点 |
|------|---------|------|------|
| A. 极简过滤器 | 顶部一排 pill，点击过滤 | 实现简单 | 无仪式感，不帮用户"进入状态" |
| B. 心理空间切换 | 切换时有色调/氛围变化，语气微调 | 用户感受到身份切换 | 实现复杂 |
| C. 上下文面板 | 切换后展示角色摘要 | 直接解决"一秒进入状态" | 多一屏信息 |
| D. 认知模式优先 | 不按角色，按深度工作/浅层/沟通分批 | 更科学 | 过于激进 |
| **E. 混合方案** | **B + C 结合** | **兼顾体验和简洁** | 需要精心平衡 |

**决策 ✅**：采用 **方案 E（混合方案）**——

1. **角色定义**：用户可定义 2-N 个角色，每个角色有名称 + 可选色调/图标
2. **AI 自动归类**：流中的内容和任务由 AI 自动关联到角色，用户可修正
3. **切换交互**：左侧栏放置角色切换器，一次点击完成；切换时有轻微色调过渡
4. **角色着陆卡片**：切换后短暂展示上下文摘要
5. **全视图生效**：角色切换影响所有视图
6. **默认"全部"**：不切换角色时展示所有内容

## 辩论 #9：AI 功能的工程可行性

**论点 A（理想主义派）**：AI 是核心竞争力，应该尽可能智能化。

**论点 B（工程现实派）**：
1. LLM 不会跨会话学习——"学习"只能靠把行为历史塞进 prompt 上下文
2. 后台自动 AI 调用有成本和可靠性问题
3. "学习引擎"的复杂度被严重低估
4. AI 黑盒与宪法冲突——设计宪法第四条是"AI 提议，人类决定"

**综合决策 ✅**：**AI 功能分层降级 + AI 操作审计**——

### AI 功能分层

| 层级 | 内容 | 实现方式 | 版本 |
|------|------|---------|------|
| L0: 纯规则 | 此刻推荐排序、DDL 提醒、活跃时段统计 | 纯代码逻辑，不调 LLM | v1 |
| L1: 用户触发 LLM | 魔法按钮（任务提取/拆分/整理/周回顾） | 按需调 API，结果需用户确认 | v1 |
| L2: 后台轻量 LLM | 流条目自动打标签/归类 | 批量处理，**草稿态**待确认 | v1.5 |
| L3: 统计学习 | UserProfile 中的行为模式 | 纯数学统计，周期性计算 | v2 |
| L4: LLM 增强洞察 | 行为洞察文案、个性化推荐理由 | 统计结果喂给 LLM 生成自然语言 | v2 |
| L5: 高级学习 | 回避模式识别、自动策略调整 | 需要足够数据积累 | v3+ |

### AI 操作审计

设计宪法第四条的工程保障——所有 AI 操作留痕、可审查、可撤销。

## 辩论 #10：AI 操作确认——草稿态 vs 直接生效

**决策 ✅**：**早期全部草稿态，后期可配置**——

- v1/v1.5：所有后台 AI 操作产出**草稿态建议**，不直接修改数据
- v2+：设置中增加"AI 自动生效"开关，默认关闭

## 辩论 #11：任务模型重构——步骤列表 vs 自由内容 + 子任务

**决策 ✅**：采用 **自由 body + 子任务**——

1. **Task.body**：自由 Markdown 文本
2. **Task.subtaskIds**：子任务 ID 列表，子任务是完整的 Task 对象
3. **Task.parentId**：如果自己是子任务，指向父任务
4. **去掉 steps/progress**：这两个字段废弃
5. **进度**：通过 `completedSubtasks / totalSubtasks` 计算
6. **body 中的 checkbox**：`- [ ]` 可渲染可勾选，但只是 body 内的文本，不是独立步骤

**新设计原则**：
> "不过度教用户做事，让用户以最自然的方式做他们想做的。"
