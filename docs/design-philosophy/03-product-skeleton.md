# 产品骨架：三个界面 + 一个引擎

> 这是产品的结构性设计雏形。还不是线框图，是"这个产品由哪几个部分组成、它们之间怎么流转"的骨架。

---

## 总体架构

```
用户的脑子
   │
   │ "想到什么写什么"
   ▼
┌──────────┐    AI 默默提取     ┌───────────┐
│  流       │ ──────────────→  │  结构化    │
│ (Stream)  │    任务/事件/     │  任务池    │
│           │    情绪/笔记      │           │
└──────────┘                   └─────┬─────┘
                                     │
                           AI 推荐 + 排序
                                     │
                                     ▼
                              ┌──────────┐
                              │  此刻     │ ← "现在做这个"
                              │  (Now)    │    接受 / 拒绝(学习)
                              └────┬─────┘
                                   │
                            开始 → 执行 → 完成
                                   │
                                   ▼
                              ┌──────────┐
                              │  承诺板   │ ← DDL + 提交 + 记录
                              │  (Board)  │    延期需要理由
                              └──────────┘
                                   │
                                   ▼
                              ┌──────────┐
                              │  学习引擎 │ ← 观察一切，适应你
                              │ (Engine)  │    不评判，只调整
                              └──────────┘
```

## 界面一：流（Stream）

### 定位
用户的思维倾倒场。不是"任务输入框"，是"想到什么写什么"的地方。

### 核心交互

```
┌─────────────────────────────────────────┐
│                                         │
│   [在想什么就写什么...]          [发送]  │
│                                         │
│   # 标签   📎 附件   🪄 魔法           │
└─────────────────────────────────────────┘
│                                         │
│  15:32  想吃火锅了                       │
│  15:28  论文第三章还没开始写 得抓紧了     │
│  14:55  导师说周五前要交初稿             │
│  14:30  [图片] 这个排版可以参考          │
│  ...                                    │
```

### 关键规则
- 输入零摩擦：打字 → 回车，完成
- 标签可选（#标签 语法）
- 附件可选（图片、链接、文件）
- AI 在后台自动提取任务、识别日期、关联项目——用户可以完全不管
- 流中的内容永远按时间线排列，最新的在上面
- 支持回看历史，但不是主要使用场景

### 🪄 魔法按钮在流中的行为
- 在空输入框上按：生成建议（"我看了你最近的情况，建议你现在做 X"）
- 选中某条内容后按：把这条模糊想法结构化（提取任务、建议DDL、拆子步骤）
- 积累大量内容后按：整理总结（"这 15 条里真正需要做的是这 4 件"）

## 界面二：此刻（Now）

### 定位
整个产品最重要的界面。回答唯一一个问题："你现在立刻应该做什么？"

### 默认开屏界面。

### 核心交互

```
┌─────────────────────────────────────────┐
│                                         │
│              论文第三章                   │
│         写出第一段的草稿就行              │
│                                         │
│        DDL: 周五 (还剩 2 天 4 小时)       │
│                                         │
│    ┌──────────┐     ┌──────────┐        │
│    │  开始做   │     │  不想做   │        │
│    └──────────┘     └──────────┘        │
│                                         │
│    为什么是这个：最紧迫 + 你已经拖了3天   │
│                                         │
│    🎲 随机选一个                         │
│                                         │
└─────────────────────────────────────────┘
```

### 交互流程

**点"开始做"→ 进入专注模式**
- 计时器开始
- 界面简化，只显示当前任务的详情/子步骤
- 可以暂停、完成、或放弃

**点"不想做"→ 轻量反馈**
```
为什么不想做？（点一个就行）

○ 现在没条件做这个
○ 太大了不知道从哪开始
○ 就是不想
○ 有更急的事
```
选完后自动推荐下一个。系统记录选择用于学习。

**连续拒绝 3 次 → 温和干预**
> "看起来你现在不在状态。要不要设个 30 分钟后的闹钟？还是今天就算了？"

**🎲 随机模式**
从今天可做的任务里纯随机选一个。没有算法，没有优先级，就是骰子。

### 划走规则
- 可以从"此刻"划走进入"流"或"承诺板"
- 划走 3 次后，下次回到 App 强制显示"此刻"一次
- 强制时语气温和："嘿，先看一眼？看完了再去别处"
- 用户可以选"现在不是工作时间"→ 系统放行，设定下一次提醒时间

## 界面三：任务视图（Board）

### 定位
DDL 驱动的任务执行追踪。展示所有活跃任务，DDL 任务按紧迫度高亮。

### DDL 硬度分级

| 类型 | 来源 | 能改吗 | 改的代价 |
|------|------|--------|----------|
| 硬 DDL | 外部（导师/工作/系统识别） | 不能改 | — |
| 承诺 DDL | 用户自己设定 + 确认 | 能，但要"申请延期" | 写一句原因，记录在案 |
| 软 DDL | AI 建议的 | 随意改 | 无代价 |

### "申请延期"机制
- 点击后弹出：延到什么时候？为什么延？（一句话就行）
- 系统记录延期历史
- 同一任务第 N 次延期时提醒

### "提交成果"机制
- 有产出物的任务：附上文件/链接/截图
- 无产出物的任务：写一句话"我做了什么"
- 提交后标记时间戳 + 准时/迟交状态
- 提交的仪式感 > 简单勾选

## 引擎：统计分析 + LLM 文案层

### 定位
后台运行，不直接呈现为界面，但驱动所有界面的行为。

> **工程降级说明**（见 `02-design-debates.md` 辩论 #9）：原设计中的"学习引擎"从 AI 驱动降级为**统计模块 + LLM 文案层**。用户体验不变，但工程上可行。

### 底层：纯统计模块（不依赖 LLM）

| 统计指标 | 数据来源 | 计算方式 | 用途 |
|----------|----------|----------|------|
| 活跃时段分布 | app_opened、focus_started 事件 | 按小时统计频率 | 推荐时机 |
| 平均启动延迟 | recommendation_accepted 到 focus_started | 时间差平均值 | DDL 校准 |
| 平均专注时长 | focus_started 到 focus_completed | 时间差平均值 | 任务拆分参考 |
| DDL 偏差率 | ddl_met vs ddl_missed 比例 | 频率统计 | DDL 建议 |
| 拒绝原因分布 | recommendation_rejected 的 reason 字段 | 分类计数 | 推荐调优 |
| 任务类型偏好 | 接受的推荐的 tag 分布 | 频率排序 | 推荐排序权重 |

### 推荐排序（L0，纯规则，不依赖 LLM）

```
score = w1 × DDL紧迫度 + w2 × 任务年龄 + w3 × 时段匹配度 - w4 × 连续拒绝次数
```

### 核心原则
- 统计是确定性的：不依赖 LLM 的概率性输出，可测试、可复现
- 调整是渐进的：不会因为一次拒绝就大幅改变策略
- 洞察是温和的：不评判，只描述模式，让用户自己得出结论
- LLM 是锦上添花：没有 API Key 时，功能不降级

## 角色/项目切换

> 辩论过程见 `02-design-debates.md` 辩论 #8。

### 核心理念

角色不是过滤器，而是身份。切换角色 = 切换心智模式。

### 角色着陆卡片

切换角色后，短暂展示一张上下文摘要卡片：

```
┌───────────────────────────────────────────┐
│  🎓 研究生                                │
│                                           │
│  上次：在写论文第三章的第一段草稿          │
│  待办：3 件事，其中 1 件今天到期           │
│  距上次切入：2 天前                        │
│                                           │
└───────────────────────────────────────────┘
```

## 界面四：任务详情面板（Task Detail Panel）

### 定位
点击任何任务卡片后，从右侧滑出的详情面板。

### 关键能力
- 内联编辑标题
- 角色 pill 可点击切换
- 状态下拉选择（收件箱/活跃/今天做/已完成/已归档）
- DDL 日期选择 + 类型（硬性/承诺/弹性）
- body 编辑器：自由文本，支持 Markdown checkbox
- 子任务管理：新建、勾选完成、提取为独立任务
- 历史：延期记录、提交记录
- 删除任务（二次确认）

---

## 设计原则（补充）

> **"不过度教用户做事，让用户以最自然的方式做他们想做的。"**

- 输入应该是自由的，结构是涌现的而非强制的
- 不要求用户记住特殊语法才能触发功能
- 提供工具让用户组织，但不强制使用
