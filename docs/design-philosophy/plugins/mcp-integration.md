# MCP 集成（`mcp-integration`）— 设计原则

> **模块 ID**：`mcp-integration`  
> **作用**：在受控前提下，让外部智能体（Cursor、Claude Desktop 等）通过 MCP 协议读写用户的任务、流、角色等数据。

本文与仓库根目录 `skills/SKILL.md`（面向模型 / Agent 的操作说明）互补：**本文写产品与设计原则**，技能文件写工具与调用约定。

---

## 一、定位与边界

1. **外部执行系统，不是「远程全权限脚本」**  
   MCP 连接的是与用户同一套数据模型（任务、流、角色、DDL 等），行为应符合产品隐喻：辅助记录与推进执行，而不是替用户「擅自清空人生」。

2. **用户拥有总开关**  
   关闭「MCP 集成」后，服务端 **不得** 再对外提供 MCP 能力（实现上通过设置项门闩，见下节）。这是信任与合规底线。

3. **权限由用户分层配置**  
   - **等级**：Read / Create / Full — 低等级不能调用高等级工具。  
   - **角色 ACL**：可限制智能体只能看到部分角色的数据。  
   - **单工具开关**：用户可禁用个别工具，列表以运行时 `tools/list` 为准。

4. **与认证模式的关系**  
   - 多用户 / 单用户服务端：通常需要 `Authorization: Bearer <token>`。  
   - 桌面嵌入式本地服务（`AuthMode::None`）：本机回环访问，不依赖用户 token，但仍受 **模块开关 + 上述用户侧 MCP 配置** 约束。

---

## 二、服务端门闩（必须遵守）

- 设置键：`module:mcp-integration:enabled`（与前端 `module:mcp-integration:enabled` 同一语义，由同步/API 写入用户设置命名空间）。
- MCP 处理逻辑在响应前应检查该开关；关闭时端点应对外表现为不可用（与产品「关闭即消失」一致）。

---

## 三、工具设计的哲学

1. **优先给「全貌」，避免拉全量**  
   提供 `get_overview` 等聚合能力，避免一上来 `list_tasks` 无分页拉满，加重服务端与用户心智负担（与技能文件中的反模式一致）。

2. **破坏性操作属于 Full**  
   删除任务、删改角色等必须与最高权限等级绑定，且技能侧应强调确认与反模式（不未确认就删除）。

3. **与「插件关闭」一致**  
   若产品层还有其他模块门闩（例如仅某功能可用时），MCP 工具行为应与 UI 一致，不出现「界面上关了，API 仍能操作」的割裂。

---

## 四、客户端与 URL

- **Web / 远程 API**：MCP URL 一般同源或 `settings` 中配置的 API 基址下的 `/api/mcp`。  
- **Tauri 桌面**：本地嵌入式 HTTP 服务固定本机端口（实现见 `packages/web/src-tauri`），配置展示与复制 JSON 时应使用实际可访问的 base URL；无 token 时不在配置中强加无意义的 `Authorization` 头。

---

## 五、文档与技能同步

- 对外工具列表、权限模型变更时：同时更新 **服务端路由**、**设置 UI 文案**、**`skills/SKILL.md`**，避免文档与真实 `tools/list` 脱节。

---

## 六、反模式（设计层）

- 把 MCP 当作「绕过产品哲学的后门」：例如默认全开 Full、无 ACL、无模块门闩。  
- 在错误信息或返回体中泄露内部实现细节或敏感路径。  
- 为「方便调试」长期在非本机环境使用 `AuthMode::None` 且对公网暴露。
