# 工作线程运行时（`work-thread-runtime`）— 产品设想

> 这是 Work Thread 2.0 的产品锚点文档。  
> 它回答的不是“线程页长什么样”，而是“为什么需要线程运行时、它和现有产品骨架怎么配合、哪些东西必须成立，哪些东西暂时不做”。

---

## 一、定位

Work Thread 不是 AI 协作面板，也不是另一种任务列表。

它的核心定位是：

**为单个执行者提供一个可随时恢复的上下文环境。**

更具体地说：

- `Stream` 负责低摩擦捕获
- `Task` 负责承诺、DDL、交付
- `Work Thread` 负责维持“我正在推进哪条工作线、为什么停在这里、回来时先看什么、下一步先做什么”

因此，Work Thread 是现有产品里的**运行时层**，而不是替代 `Stream / Task / Now` 的新主干。

---

## 二、它要解决的问题

当前产品已经能解决“先记下来”和“现在做哪件事”的问题，但还不够好地解决下面这类真实工作场景：

1. 用户在推进的是一个持续中的问题，而不是一条孤立任务。
2. 用户会频繁中断，然后回来时忘记自己停在哪里。
3. 用户需要保存的不只是待办事项，而是当前的上下文环境。
4. 很多工作并不是“没做”或“做完”，而是“在等”“卡住”“先挂起”。

Work Thread 2.0 要解决的，不是“怎么让 AI 帮我做更多”，而是：

**怎么让用户在切走后还能顺畅地重新进入工作。**

---

## 三、为什么采用 agent-like 表现形式

Work Thread 2.0 会借鉴 agent 产品的外在表现形式，但不把 AI 当成中心。

要借鉴的是这些结构：

- `session`：一条持续中的工作线，而不是一次性的对话
- `append-only event log`：能回看“做过什么、为什么停”
- `pinned context`：关键上下文要能固定呈现
- `checkpoint / resume`：离开时保存恢复点，回来时优先恢复
- `working set`：先看当前最需要的 3 到 7 条上下文，而不是全量历史

这类形式优越的原因不是“像聊天”，而是它天然适合长任务、切换和恢复。

对人类来说，这意味着：

- 不用把所有信息都留在脑子里
- 不用每次回来重新扫全文
- 不用靠意志力维护“当前上下文”

---

## 四、产品骨架中的关系

### 1. Stream = 捕获层

`Stream` 继续承载：

- 灵感
- 吐槽
- 片段
- 链接
- 中断
- 原始材料

它是事件流，不是线程本体。

### 2. Task = 承诺层

`Task` 继续承载：

- 明确动作
- DDL
- 状态
- 交付
- 子任务

它是可追踪承诺，不是上下文环境。

### 3. Work Thread = 运行时层

`Work Thread` 承载：

- mission
- 当前状态
- resume card
- working set
- waiting conditions
- linked tasks
- 决策记录
- 时间线

它负责回答：

- 这条工作线在推进什么？
- 为什么现在停在这里？
- 再回来时先做什么？
- 当前最该优先加载哪些上下文？

### 4. Now = 调度入口

`Now` 仍然回答“现在做什么”，但不只从 task 里选。

Work Thread 2.0 上线后，Now 的来源会变成双通道：

- thread-first
- task fallback

也就是：

“回到这条线程，并先做这一步”

会成为和“现在做这个任务”同等的一等推荐对象。

---

## 五、核心对象

### 1. Mission

Mission 是线程存在的理由。

它不是标题，也不是任务名，而是一句能描述：

- 这条线到底在推进什么
- 做到什么程度才算完成

的表达。

### 2. Resume Card

Resume Card 是线程的热启动面。

它至少包含：

- 我停在这里
- 下一步先做什么
- 别忘了的前提/护栏
- 当前等待摘要

离开线程时要尽量留下它，回来线程时优先展示它。

### 3. Working Set

Working Set 不是全部材料，而是本次恢复最值得先看的那几项。

它通常来自：

- 当前最相关的 context item
- 最近的决策
- 正在执行或刚创建的 task
- 当前 waiting 条件对应的材料

### 4. Waiting Conditions

线程不是简单的“进行中 / 完成”，而是可能处在：

- 等人回复
- 等文件
- 等时间点
- 等工具准备好
- 等外部结果

这类状态必须被显式建模，否则用户只能靠脑内记忆维护。

### 5. Interrupts

工作中冒出来的新东西不一定要立刻打断主线。

Interrupts 用来暂存：

- 新灵感
- 新链接
- 突然想到的事项
- 外部刺激触发的分支

它们可以稍后再并入主线程、转成 task，或单独升格为新线程。

### 6. Scheduler Policy

调度器不是老板，而是辅助系统。

它的职责是帮助单核 worker 做轻量调度，而不是替用户决定人生。

---

## 六、线程状态机

Work Thread 2.0 采用下面的状态：

- `running`
- `ready`
- `waiting`
- `blocked`
- `sleeping`
- `done`
- `archived`

这些状态的区别必须清楚：

- `running`：当前正在推进
- `ready`：可继续推进，等待调度
- `waiting`：在等外部条件成熟
- `blocked`：知道下一步，但当前做不了
- `sleeping`：主动暂时搁置
- `done`：线程目标已完成
- `archived`：历史归档，不再参与调度

这里最关键的是：

**“没在做”不等于“做不了”，更不等于“做完了”。**

---

## 七、调度器定位

### 1. 谁是 worker

用户不是 manager，也不是 orchestrator-first 的上帝视角控制台。

在这个系统里，更贴近现实的隐喻是：

- 用户 = 单核 worker
- 系统 = 调度器
- Thread = 可恢复执行上下文

### 2. 调度器不是老板

调度器负责减少切换成本和调度负担，但不能强夺控制权。

默认要支持三档策略：

- `manual`
- `coach`
- `semi_auto`

#### `manual`

- 不主动安排
- 主要负责 checkpoint / resume
- 适合想完全自己掌控节奏的人

#### `coach`

- 温和推荐当前值得恢复的线程
- 不强推，不自动接管
- 这是默认建议档

#### `semi_auto`

- 默认优先把最值得恢复的线程推到 Now
- 仍然保留用户确认和跳过能力
- 适合更信任系统调度的人

### 3. 调度原则

第一阶段调度器只做轻量规则，不做复杂智能：

- `running` 线程优先保护连续性
- `ready` 线程根据恢复价值排序
- `waiting / blocked / sleeping` 不直接派发
- `meta`、`infrastructure` 线程默认降权
- 有明确 `resume card` 和 `next step` 的线程优先

---

## 八、AI 的边界

AI 在这套系统里是 sidecar，不是本体。

### AI 可以做的事

- 帮忙整理上下文
- 草拟 resume card
- 提取候选 next steps
- 压缩长线程

### AI 不能成为成立前提的事

- 没有 AI 就无法创建线程
- 没有 AI 就无法 checkpoint
- 没有 AI 就无法 resume
- 没有 AI 就无法调度到 Now

**关闭 AI 后，这整套线程运行时必须依然完整。**

---

## 九、默认界面形态

### 1. 线程列表页：Thread Board

不是普通列表，而是按状态分栏的线程板：

- Running
- Ready
- Waiting
- Blocked
- Sleeping
- Done

这个界面要让用户一眼知道：

- 现在在跑什么
- 哪些线程随时可捡起来
- 哪些线程其实在等
- 哪些线程是卡住了

### 2. 线程详情页：Resume-first Workspace

默认顺序不是“全文文档”，而应该是：

1. Mission
2. Resume Card
3. Main document
4. Working Set / Waiting / Timeline / Linked Tasks

也就是说，线程详情页首先是恢复工作面，其次才是文档容器。

---

## 十、明确非目标

第一阶段明确不做以下事情：

- 不做多 agent 编排
- 不做 server 侧新 API
- 不把 Stream 改成线程本体
- 不替代现有 Task Board
- 不把 Now 变成纯线程产品
- 不让 AI 接管线程调度

这些都可能是未来方向，但不属于 Work Thread 2.0 的第一阶段目标。

---

## 十一、验收标准

当下面这些感受成立时，Work Thread 2.0 才算方向正确：

1. 用户能把一条工作线当成“上下文容器”而不是“另一篇笔记”。
2. 用户切走后，再回来时能明显更快恢复状态。
3. 用户能清楚区分“现在能做”“在等”“被卡住”“先挂起”。
4. 关闭 AI 后，线程系统仍然自洽、可用、不会残废。
5. `stream / task / now / thread` 四者分工清楚，不相互替代。

如果做完后仍然像“带 AI 的线程笔记页”，那就说明方向做偏了。
