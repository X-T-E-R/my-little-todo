# 用户画像与行为分析

> 本文件基于目标用户群体的调研与真实使用数据推导而来。

---

## 目标用户群体特征

通过对典型用户的 todo 工具使用数据分析，得出以下关键发现：

### 使用数据分析

| 分类 | 占比特征 | 分析 |
|------|----------|------|
| 生活记录/日志 | 最高频 | 用户的"todo 工具"实际上是生活记录场 |
| 随想/非任务 | 约 1/4 | 大量内容不是任务，而是思维碎片 |
| 已完成任务 | 中等 | 用户有完成能力，不是"做不了事的人" |
| 待办任务 | 中等 | 积压存量对比完成数并不算失控 |
| 进行中任务 | 很少 | 同时进行中的事不多 |
| 紧急任务 | 极少 | 情绪化标记——紧迫感是真实的驱动力 |
| 失败/搁置 | 少量 | 对自己诚实，会主动标记失败 |

**关键发现**：用户的实际使用以"随想流/生活记录"为主，任务管理是次要功能。**工具对用户的真实价值是"思维倾倒场"，任务管理是附带的。**

### 标签系统分析

用户的常用标签呈现出极强的个人化和情绪化特征：

- **功能性标签**：本周要干的、急迫、地点
- **情绪性标签**：犹豫、迷茫、灵感
- **生活性标签**：食物、运动
- **元认知标签**：思考、冷知识、金句
- **兴趣标签**：AI、DIY

**关键发现**：用户不会遵守预设的分类体系。标签是自发生长的，混合功能/情绪/生活。产品不应强制标签规范化。

## 核心行为特征

### 1. 启动困难

用户需要外部推力来从"静止"切换到"行动"。障碍的本质不是"不知道做什么"，而是"无法从静止状态切换到行动状态"。任何外部推力——即使是随机的——都比自我启动容易。这是执行功能层面的特征。

### 2. DDL 驱动型

外部约束（deadline、交作业）能激活高效执行模式。没有约束时进入"自由漂浮"状态——不是懒，是缺少启动信号。

### 3. 计划逃逸

传统 todo 的"按计划执行"范式完全不适合目标用户。任何要求"按计划走"的设计都会被抛弃。产品必须容忍、甚至拥抱"不按计划"的行为。

### 4. 备忘有价值，执行无跟踪

把想法外化本身就有心理价值（认知卸载）——即使永远不再看。这意味着"记录"和"执行"在产品中应该是可以完全解耦的两个功能。

### 5. 对自我强制有矛盾心理

用户知道自己需要被推，并且能接受"被强制"。但前提是这种强制来自"为我好"而不是"控制我"。产品的强势应该有温度。

### 6. 希望系统从行为中学习

用户把"不服从建议"视为一种沟通方式而非失败。系统应该把用户的每一个行为（包括拒绝、拖延、跳过）当作数据，而不是错误。

## 用户画像总结

```
┌─────────────────────────────────────────────┐
│                                             │
│  人格类型：P 型（灵活、自发、截止日期驱动）   │
│  核心困难：启动（不是记忆、不是能力）         │
│  有效激活：外部压力、DDL、他人催促            │
│  无效手段：自我计划、每日清单、打卡            │
│  记录习惯：高频随想流，混合一切               │
│  对工具态度：用则深度用，弃则彻底弃            │
│  对强制态度：接受有温度的强制推力              │
│  对自己：诚实、有自嘲精神、愿意被理解         │
│                                             │
└─────────────────────────────────────────────┘
```

## 多端使用场景

| 场景 | 设备 | 典型行为 | 设计要点 |
|------|------|----------|----------|
| 桌面办公 | PC (Tauri) | 长时间工作，详细编辑 | 完整功能，内嵌后端 |
| 外出速记 | 手机 (PWA) | 快速记录想法，检查任务 | 触摸优化，离线缓存 |
| 他人电脑 | 网页端 | 临时查看/记录 | 登录后直接用 |
| 局域网共用 | PC 做服务器 + 手机访问 | 家庭/办公室内同步 | 开启 LAN 访问 |

**关键发现**：用户需要"随时随地记录"的能力，但"深度整理"主要在 PC 上完成。移动端的核心价值是"不丢失那一刻的想法"。

## 产品设计含义（从画像推出的铁律）

1. **不能依赖用户"每天打开"来维持系统运转** → 系统必须在用户不看的时候也能自我维护（AI 自动整理、自动归档）
2. **不能要求用户"按计划执行"** → "计划"是建议不是合同，违约无后果
3. **必须提供外部推力** → "此刻"视图 + 开屏强制 + 划走拉回
4. **必须拥抱混乱输入** → 思维流 > 结构化表单
5. **必须把不服从当数据** → 学习引擎的核心原料
6. **DDL 是核心机制，不是辅助功能** → 承诺板的设计权重要极高
7. **多端无缝切换** → 数据通过统一 API 同步，任何端记录的内容其他端立刻可见
